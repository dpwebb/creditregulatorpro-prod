# Hard-Code Account Number Exclusion & Fix Balance Mapping

## Summary
TransUnion (and Canadian consumer disclosures in general) never show full account numbers — this is by design, not a data quality issue. The platform currently flags empty account numbers as an "INCOMPLETE_DATA" creditor violation (Metro2 §4.2), generating false positives for every tradeline. This plan removes that false-positive check permanently and also fixes the `balance` column never being populated during ingestion.

## Files to Modify

### 1. `helpers/regulationInfractionScannerCreditor.tsx`
- **Remove check #4** ("Incomplete Account Data — Metro2 §4.2") entirely. Canadian bureau consumer disclosures always mask/redact account numbers — flagging this was always a false positive. This check should not exist for a Canada-only platform.

### 2. `helpers/ingestTradelinePersistence.tsx`
- In the `tradelineData` object, add `balance: parsedTradeline.balance` alongside the existing `currentBalance: parsedTradeline.balance` so both DB columns get populated.
- Fix existing TS errors on lines 52-53: wrap `candidate.currentBalance` in `Number()` for the `Math.min`/`Math.max` calls.

## Files to Create
None.

## Approach
1. Remove check #4 from `regulationInfractionScannerCreditor.tsx`.
2. Add `balance` mapping in `ingestTradelinePersistence.tsx` and fix TS errors.
3. Backfill existing tradelines via SQL: `UPDATE tradeline SET balance = current_balance WHERE balance IS NULL AND current_balance IS NOT NULL`.
4. Delete any existing infraction/compliance records generated by the false "INCOMPLETE_DATA" / account number violation.

## Risks & Considerations
- Removing the check is safe — it was always a false positive for Canadian disclosures.
- The `balance` backfill is a simple data copy with no risk.
- No frontend/UI changes needed.
