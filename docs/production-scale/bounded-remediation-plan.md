## Boundary Rules

This remediation plan preserves the current application structure, current operational model, and current Vitest/script-based test framework. Remediation work must patch existing files, services, workflows, or scripts where practical. Thin helper services and small validation scripts are allowed only when they reduce risk without replacing existing subsystems.

Do not replace the parser, queue, packet, evidence, deploy, migration, or storage systems wholesale. Do not introduce live external provider calls. Do not require manual browser testing, manual deploy testing, manual queue draining, or human interaction to mark a task complete.

No task may be marked complete unless automated evidence exists. Evidence must be generated by checked-in Vitest tests, static workflow checks, dry-run scripts, simulated harnesses, or other repeatable repository commands. Dashboard output, stale promotion packs, simulated proof, or passing happy-path tests alone must not be represented as production certification. If a production-ready claim cannot be proven by automation, the output must continue to say `CERTIFYING:false`.
